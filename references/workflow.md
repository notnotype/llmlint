# 完整流程详解

本文档解释 `llmlint` 的 6 步文本润色流程，包括 CLI 候选定位、LLM 全文审查、快速审查评分、用户审批和修复报告。

## 流程概览

```
用户输入 → CLI 静态检查 → LLM 深度审查 + 快速审查评分 → 生成修复计划（用户审批）→ 执行修复 → 生成报告
```

## 步骤 1：获取输入文本

### 目标

将用户提供的文本转换为可检查的文件路径。

### 实现细节

**识别输入类型**：
```typescript
if (用户消息包含文件路径) {
    // 识别模式：
    // - "润色 manuscript/chapter-01.md"
    // - "帮我看看 .agent/test.md 是否自然"
    文件路径 = 提取的路径
} else {
    // 用户直接粘贴文本
    文件路径 = ".agent/polish-input.md"
    write(文件路径, 用户粘贴的文本)
}
```

### 边界情况

**文件不存在**：
- 检查文件路径是否有效。
- 如果无效，询问用户确认路径。

**路径歧义**：
- `manuscript/chapter-01.md` 可能指多个 Project Workspace。
- 优先按当前 Project Workspace 解析。

**空文本**：
- 提示用户提供有效内容。

### 错误恢复

- 文件读取失败 → 提示用户检查路径和权限。
- 编码问题 → 提示确保 UTF-8 编码。

## 步骤 2：CLI 静态检查

### 目标

快速识别可以稳定定位的候选问题，生成可供 Agent 复核的报告。

默认配置会加载 `builtin/default`。它包含 R18/成人词汇规则，内置文件按 namespace 拆分只是资产维护结构；用户若在配置中关闭了 `vocabulary.r18` namespace，Agent 不应再按该 namespace 提出候选。

### 实现细节

**执行命令**：
```bash
bun .nbook/agent/skills/llmlint/bin/llmlint.ts check <文件路径>
```

候选过多时先看中高等级：

```bash
bun .nbook/agent/skills/llmlint/bin/llmlint.ts check <文件路径> --min-level medium
```

小文件或人类阅读时显示完整命中行：

```bash
bun .nbook/agent/skills/llmlint/bin/llmlint.ts check <文件路径> --show-lines
```

**解析输出**：
CLI 默认输出先按 high / medium / low 分段，再按 regex rule 分组。每条命中包含规则 ID、namespace、ruleset 来源、行号、闭区间列范围、命中文本和 action 建议：
```text
medium (1 problem)

filler-word-actually [filler] (无意义填充词)
  来源：builtin/default；级别：medium
  1:9-10  match: 其实

  1 occurrence. 建议删除。
```

注意：CLI 只负责定位候选文本，不负责决定是否修复。所有 regex 命中项都需要结合上下文复核。

### 边界情况

**CLI 执行失败**：
- bun 未安装 → 提示用户安装 bun。
- 文件路径错误 → 检查路径是否正确。
- 规则文件损坏 → 检查 JSON 文件是否有效。

**无问题**：
- CLI 输出 `✓ No problems found`。
- 仍然进入步骤 3 的快速审查评分，确认全文是否有语义级问题。

**问题过多**：
- `check` 默认只展示 `review: agent` 的命中（需要 Agent/LLM 处理的）；破折号、比喻、泛词形副词等已默认归入 `human` 桶不展示。需要查看时用 `--review human` 或 `--review all`。
- 超过 50 个候选时，优先使用 `--min-level medium` 或 `--min-level high` 降噪，再分段处理。
- `--min-level` 和 `--review` 只隐藏展示结果，不改变规则配置；需要关闭某类规则时使用 `llmlint.config.ts` 的 `namespaces` 或 `rules`。

### 错误恢复

- CLI 退出码非 0 不一定代表命令失败，可能只是发现 high 级别问题。
- 输出解析失败时，直接展示原始 CLI 输出给用户。
- 超时时建议分段检查。

## 步骤 3：LLM 深度审查 + 快速审查评分

### 目标

读取 LLM 规则，对 regex 命中项与 LLM 语义规则进行上下文审查，再用快速审查清单和 50 分评分补足整体风格判断。

### 获取 LLM 规则

必须先执行：

```bash
bun .nbook/agent/skills/llmlint/bin/llmlint.ts show-llm-rules
```

这一步仍属于步骤 3，不单独拆成新步骤。`show-llm-rules` 的输出是本轮全文语义审查的规则来源。

### 审查流程

1. 复核 CLI regex 命中项：
   - 先根据行列范围和 `match` 定位候选；需要判断语境时继续读取原文前后段落。
   - 如果输出用于人类快速阅读或小文件审查，可以重新运行 `check --show-lines` 查看 `<mark>` 标出的完整命中行。
   - 判断候选是否真的影响自然度、密度或可信度。
   - 给出建议修复、建议保留或需要用户确认。

2. 逐条审查 LLM rules：
   - 对 `show-llm-rules` 输出的每条规则主动阅读全文。
   - 不等待 CLI 给出 LLM 命中项。
   - 找到候选后读取上下文，应用该规则的判断标准和示例。
   - 如果没有候选，也在计划中说明未发现明显问题。

3. 执行快速审查：
   - 检查全文是否存在 stop-slop 风格的问题：铺垫、反转、过度解释、抽象总结、隐藏行动者、懒惰绝对词、金句感、节奏单调。
   - 快速审查可以发现未被 CLI 或 LLM rule 精确覆盖的局部问题。

4. 给出 50 分评分：
   - Directness / Rhythm / Trust / Authenticity / Density 各 1-10 分。
   - 写明总分和处理等级。

### 判断标准

**Regex Rules**：
- High：强烈建议修复，但仍需确认上下文是否合理。
- Medium：读取前后 2-3 行判断；对话和口语场景可能保留。
- Low：默认保留，除非明显影响阅读。

**LLM Rules**：
- 严格遵循 `show-llm-rules` 输出中的判断标准。
- 参考示例，但不要机械套用示例。
- 不确定时归入“需要用户确认”。

**快速审查清单**：
- 删除测试：删掉这个词/短语/句子后，信息是否几乎不变？
- 具体性测试：抽象判断能否换成具体动作、事实、主体或结果？
- 主体测试：句子是否说明谁做了什么，而不是让“数据、文化、局面、决定”自己行动？
- 信任测试：是否在解释读者已经能理解的东西？
- 节奏测试：连续几句的长度、结构、段尾方式是否过于相似？
- 金句测试：句子是否像可摘出来当标语，却没有承担实际信息？

### 50 分评分规则

| 维度 | 1-4 分 | 5-7 分 | 8-10 分 |
|------|--------|--------|---------|
| Directness | 大量公告、铺垫、绕弯 | 部分句子可更直接 | 多数句子直接进入内容 |
| Rhythm | 句式和段尾高度重复 | 有变化但仍有机械段落 | 长短、转折、收束自然 |
| Trust | 说教、反复解释、替读者下结论 | 偶尔过度解释 | 给足信息，让读者自己判断 |
| Authenticity | 像模板或通用助手口吻 | 有作者声音但不稳定 | 声音贴合体裁、角色和场景 |
| Density | 很多可删内容 | 局部松散 | 词句承担清晰信息 |

评分处理：
- **低于 35/50**：必须进入重点重写建议，优先处理结构、节奏和具体性。
- **35-42/50**：做局部修订，优先删除填充、重写空泛句和调整段落节奏。
- **43/50 以上**：只做轻微润色建议，避免把作者风格磨平。

### 读取上下文策略

**简单问题（filling words）**：
- 前后各 1-2 行足够。

**二元对比、负向列举、公式化设问**：
- 读取前后各 3-5 行，或完整段落。
- 判断它是否纠正真实误解、推进论证，还是只制造戏剧化。

**段落级 LLM rule**：
- 优先读取完整段落。
- 如果段落承接前后文，再读取相邻段落。

### Web 调研触发

**触发条件**：
1. 文本包含专有名词（人名、地名、作品名）。
2. 涉及科幻设定（需要确认是否是原著梗）。
3. 同人作品引用。
4. 特定领域术语。

**调研内容**：
```typescript
await invoke_agent("researcher", {
    query: "《三体》中「不是XX，而是YY」是否是原著名句"
});
```

**使用调研结果**：
- 如果是原著引用 → 保留。
- 如果是常见领域表达 → 根据上下文判断。
- 如果查无此内容 → 按文本润色规则处理。

### 决策记录格式

审查结论是给用户看的，使用 Markdown 小节、短段落和列表，不使用 JSON、YAML、TypeScript interface 等格式化数据结构。

```markdown
### 行 34:5 - 二元对比结构

原文：不是因为天气不好，而是因为路况复杂。

建议：改为“路况复杂导致延误。”

理由：前半句没有纠正常见误解，只是在制造反转；直接陈述原因更自然。
```

### 边界情况

**问题文本难以定位**：
- 正则匹配到的文本可能不唯一。
- 使用行号 + 列号精确定位。

**上下文不足**：
- 问题在文件开头/结尾。
- 尽可能读取可用的上下文。

**风格被误伤**：
- 对话、第一人称、讽刺、角色口癖、技术步骤可能故意使用某些模板句式。
- 标记为保留或需要用户确认，不要强行改。

### 错误恢复

- LLM 判断超时 → 标记为 `needsUserConfirm`。
- Web 调研失败 → 按无调研结果处理。
- 上下文读取失败 → 使用问题本身的文本。

## 步骤 4：生成修复计划（用户审批）

### 目标

生成清晰的修复计划，让用户理解和审批。

### 计划内容

**统计部分**：
```markdown
## 统计
- 强烈建议修复：X 项
- 建议修复：Y 项
- 保留不改：Z 项
- 不确定项：N 项
```

**评分部分**：
```markdown
## 快速审查评分
- Directness：7/10
- Rhythm：6/10
- Trust：8/10
- Authenticity：7/10
- Density：6/10
- 总分：34/50，需要重点重写建议
```

**修复详情**：
```markdown
### [1] 行 5:12 - 填充词：其实 ✓ 建议修复
**原文**：这个问题很复杂。其实我们可以从另一个角度来看。
**修改后**：这个问题很复杂。我们可以从另一个角度来看。
**理由**：填充词不增加实质内容。

### [2] 行 34:5 - 二元对比结构 ❓ 需要确认
**原文**：不是因为天气不好，而是因为路况复杂。
**建议修改**：路况复杂导致我们没有按时到达。
**理由**：可以直接陈述后半部分，但可能损失对比强调的效果。
**需要确认**：这里是否要保留对比？
```

### 展示方式

使用面向用户的 Markdown 直接展示修复计划。问题较少时展示完整详情；问题较多时先展示统计、评分、关键样例和不确定项，并把完整计划写入 `.agent/polish-plan.md`。

### 解析用户反馈

**识别模式**：
```
"确认" / "全部接受" / "OK"
  → 接受所有建议修复

"跳过第 3、5 项"
  → 拒绝第 3 和第 5 项修复

"第 3 项改成：路况复杂是主要原因"
  → 修改第 3 项的修复内容

"保留第 2 项的对比"
  → 将第 2 项的 action 改为 "keep"
```

### 错误恢复

- 用户反馈解析失败 → 再次询问，提供明确的选项。
- 指定的项不存在 → 提示有效的项范围。

## 步骤 5：执行修复

### 目标

安全、准确地执行文本修改。

### 修改策略

**从后往前修改**：
```typescript
const edits = decisions
    .filter((decision) => decision.action === "fix")
    .sort((a, b) => b.line - a.line);

for (const edit of edits) {
    edit(文件路径, {
        oldText: edit.oldText,
        newText: edit.newText,
    });
}
```

**为什么从后往前**：
- 修改前面的行会导致后面的行号偏移。
- 从后往前修改，前面的行号保持不变。

### 精确匹配

**oldText 定位**：
- 使用完整的匹配文本（包括前后空格）。
- 如果文本不唯一，使用行号辅助定位。

**示例**：
```typescript
// 不好：可能匹配多处
oldText: "其实";

// 好：包含上下文，唯一匹配
oldText: "这个问题很复杂。其实我们可以从另一个角度来看。";
```

### 生成新文件

默认生成 `.agent/polish-output.md`，保留原文件作为对比。只有用户明确要求时才直接修改原文件。

### 边界情况

**oldText 找不到**：
- 文件内容可能已经被外部修改。
- 重新读取文件，更新 oldText。
- 如果仍找不到，跳过该项并记录。

**newText 引入新问题**：
- 第一版不自动迭代。
- 如用户要求，可在修复后重新运行 check。

**修改冲突**：
- 两个修改涉及同一行。
- 记录冲突并提示用户确认。

## 步骤 6：生成修复报告

### 目标

提供清晰的修复总结和对比。

### 报告内容

**统计部分**：
```markdown
## 统计
- 总候选数：15
- 已修复：12
- 保留不改：3
- 修复类别：填充短语（8）、机械过渡（4）
```

**评分变化**：
```markdown
## 快速审查评分
- 润色前：34/50
- 润色后：43/50
- 主要改善：句子更直接，删除了空泛总结和重复铺垫。
```

**修复详情**：
```markdown
## 修复详情

### [1] 行 5:12 - 填充词：其实 ✓ 已修复
- **原文**：这个问题很复杂。其实我们可以从另一个角度来看。
+ **修改**：这个问题很复杂。我们可以从另一个角度来看。
```

### 展示方式

**简要版**（推荐）：
```text
润色完成。

- 总共 15 个候选问题
- 已修复 12 个
- 保留 3 个
- 快速审查评分：34/50 → 43/50

输出文件：.agent/polish-output.md
详细报告：.agent/polish-final-report.md
```

**详细版**（用户要求时）：
- 展示完整报告内容。
- 或展示几个关键修复的 diff。

### 边界情况

**所有问题都被保留**：
- 告知用户没有执行任何修复。
- 说明保留原因，避免误以为工具失效。

**修复失败项**：
- 在报告中标注失败项。
- 说明失败原因。

## 完整示例

### 用户输入

```text
用户：帮我润色这段文字

这个问题很复杂。其实我们可以从另一个角度来看。

值得注意的是，这种方法在实践中效果显著。

首先，我们需要分析问题。其次，要制定方案。最后，执行计划。
```

### Agent 执行

1. **获取输入**：写入 `.agent/polish-input.md`。
2. **CLI 检查**：发现 4 个候选问题。
3. **LLM 审查 + 快速评分**：
   - "其实" → 建议修复。
   - "值得注意的是" → 建议修复。
   - "首先...其次...最后..." → 建议修复。
   - 评分：Directness 6、Rhythm 5、Trust 7、Authenticity 6、Density 6，总分 30/50。
4. **生成计划**：展示给用户。
5. **用户确认**："确认"。
6. **执行修复**：生成 `.agent/polish-output.md`。
7. **生成报告**：展示统计、评分和文件位置。

### 修复后文本

```text
这个问题很复杂。我们可以从另一个角度来看。

这种方法在实践中效果显著。

我们先分析问题，再制定方案，然后执行。
```

## 最佳实践

### 对于 Agent

1. **读取上下文充分**：不要只看问题本身，要读取前后文。
2. **判断要有依据**：说明为什么修复或保留，引用规则定义。
3. **评分要解释**：不要只给分，说明扣分来自哪里。
4. **不确定时询问**：不要强行做决定。
5. **保护风格**：不要把有效的角色声音、讽刺和体裁格式改成通用表达。

### 对于用户

1. **分段处理**：大文件建议分段检查和修复。
2. **迭代优化**：第一轮修复后可以再检查一次。
3. **保留原文件**：修复前备份重要内容。
4. **理解规则**：阅读 patterns.md 了解常见套路化表达。

---

## 未来改进方向

1. **自动迭代**：修复后自动再检查，直到无问题。
2. **冲突检测**：检测修复引入的新问题。
3. **自定义规则示例库**：沉淀不同文体的配置片段。
4. **学习优化**：根据用户反馈调整规则权重。
