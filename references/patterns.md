# 中文文本润色模式库

本文档记录文本润色中需要重点审查的套路化表达。目标不是消灭所有固定句式，而是让文本更直接、具体、有节奏，并保留作者或角色本来需要的声音。

## 核心原则

1. **删掉填充**：开场、强调、铺垫、元叙述如果不增加信息，直接删。
2. **打破公式结构**：少用二元反转、负向列举、机械三段式、公式化设问。
3. **写出行动者**：尽量说明谁做了什么，不让“数据、文化、局面、决定”自己行动。
4. **写具体**：不用“意义重大”“影响深远”替代事实、动作、结果。
5. **信任读者**：不反复解释、不替读者下结论、不用口号式收束。
6. **调整节奏**：混合长短句，避免每段都用同一种结尾。

## 内置规则包

默认启用 `builtin/default`。它合并人工维护的 anti-ai-slop 规则与 `.agent/workspace/llmlint_rules` 中文规则样本的策展结果：同 detector 去重、同 target 的替换候选取并集。内置资产放在 `rules/` 目录下并按维护需要分层，例如 `rules/vocabulary/r18.json`；用户仍通过 `llmlint.config.ts` 控制 ruleset、namespace 或 rule id。它包含 R18/成人词汇规则；普通项目如果不希望检查这类词汇，可以在配置中设置：

```typescript
export default {
    namespaces: {
        "vocabulary.r18": "off",
    },
};
```

策展样本中的 `[可选]`、`[选开]`、`冲突` 组，以及强偏好专项来源默认 disabled；需要时通过 `namespaces` 或 rule id 显式打开。中文规则 ID 使用可读英文语义 slug，例如 `cn.vocabulary.body.skull-head`。

## Regex Detector 候选模式

这些模式可以被 CLI 稳定定位，但是否修复仍要结合上下文判断。

### 1. 填充短语

**特征**：不增加实质内容的填充词。

**常见模式**：
- "其实"、"实际上"、"事实上"
- "值得注意的是"、"需要指出的是"、"需要强调的是"
- "可以说"、"不得不说"
- "让我们"

**示例**：
- 坏例：这个问题很复杂。其实我们可以从另一个角度来看。
- 好例：这个问题很复杂。我们可以从另一个角度来看。

**保留场景**：
- 对话中的自然口癖。
- 为了缓冲强硬语气。

### 2. 机械过渡

**特征**：刻板列举或对比结构。

**常见模式**：
- "首先...其次...最后..."
- "一方面...另一方面..."

**示例**：
- 坏例：首先，我们需要分析问题。其次，要制定方案。最后，执行计划。
- 好例：我们先分析问题，再制定方案，然后执行。

**保留场景**：
- 技术文档、课程讲义、正式报告中需要清晰步骤时。

### 3. 二元对比

**特征**："不是...而是..."、"问题不是 X，是 Y" 等反转结构。

**示例**：
- 可能多余：不是因为天气不好，而是因为路况复杂。
- 改为：路况复杂导致我们没有按时到达。
- 合理使用：问题不是技术不够先进，而是沟通不够充分。（纠正真实误解）

**判断标准**：
- 保留：前后对比确实纠正了读者可能持有的误解。
- 修复：前半句只是凭空树靶，直接陈述后半句更清楚。

### 4. 喉舌式时代开头

**特征**：用宏大背景启动文章，却没有提供具体信息。

**常见模式**：
- "在当今社会..."
- "随着...的发展..."
- "众所周知..."
- "不可否认..."

**示例**：
- 坏例：随着人工智能的发展，写作正在发生深刻变化。
- 好例：今年，编辑部把三分之一的初稿交给模型处理。

### 5. 空洞抽象表达

**特征**：只给评价，不给事实。

**常见模式**：
- "具有深远的意义"
- "起到了积极的作用"
- "产生了重要影响"
- "取得了显著成效"

**示例**：
- 坏例：这次调整产生了重要影响。
- 好例：这次调整让审核时间从三天缩到半天。

### 6. 元叙述公告

**特征**：文本先宣布自己要做什么，而不是直接做。

**常见模式**：
- "下面将介绍..."
- "接下来我们将..."
- "本文将从..."
- "本节将..."

**示例**：
- 坏例：接下来我们将分析她的心理变化。
- 好例：她握住门把手，指节一点点发白。

**保留场景**：
- 教材、技术文档、报告中明确结构提示时。

### 7. 公式化设问

**特征**：用问题制造悬念，下一句立刻自问自答。

**常见模式**：
- "为什么这么说？"
- "这意味着什么？"
- "你有没有想过..."
- "试想一下..."

**示例**：
- 坏例：这意味着什么？这意味着团队必须重新分配资源。
- 好例：团队必须重新分配资源。

### 8. 强调拐杖

**特征**：用强调词替代内容本身。

**常见模式**：
- "毫无疑问"
- "显而易见"
- "毋庸置疑"
- "说到底"
- "归根结底"

**示例**：
- 坏例：毫无疑问，这是一项重要决定。
- 好例：董事会今天冻结了两个部门的预算。

### 9. 商务黑话

**特征**：用组织话术或行业套话遮住具体动作。

**常见模式**：
- "赋能"、"抓手"、"闭环"
- "沉淀"、"对齐"、"拉通"
- "落地"、"打法"、"颗粒度"
- "心智"、"生态"、"组合拳"

**示例**：
- 坏例：我们要拉通链路，形成增长闭环。
- 好例：销售和客服每天同步流失客户名单，当天回访。

**保留场景**：
- 文本面向熟悉这些术语的真实业务团队，且术语有明确内部定义。

### 10. 懒惰绝对词

**特征**：用绝对判断制造权威感，但没有限定范围。

**常见模式**：
- "所有人"、"每个人"、"大家都"
- "永远"、"从来不"、"一定会"
- "必然"、"毫无例外"、"没有人"

**示例**：
- 坏例：所有人都会被这个问题影响。
- 好例：客服组的 18 名同事每天都会遇到这个问题。

## v1 规则内容扩充（取三同类项目精华）

builtin/default 在原有规则上扩充了一批成体系的中文 regex 规则，主要取材自三个同类项目：
**shuorenhua（说人话，主来源，中文原生）**、**avoid-ai-writing（44 类 taxonomy + 通用机械规则）**、
**humanizer（Wikipedia「Signs of AI writing」概念清单）**。原则是 static 优先、覆盖优先、容忍误差。

按 shuorenhua 的 Tier 模型映射到 `level` / `review`：
- **Tier 1（默认替换、AI 高频）→ `review: agent`**：正文默认展示。
- **Tier 2/3（同段聚集 / 全文密度才算）→ `review: human`**：regex 无法判密度，降到 human 桶避免逐次误杀。

新增命名空间（agent 桶，默认展示）：
- `opening.cliche` 开场套话：值得一提的是、不难发现、众所周知、让我们一起来看看、在当今这个…的时代。
- `inflation.significance` 渲染性强调：深刻的影响、至关重要、前所未有、颠覆性、范式转移、令人瞩目、发人深省。
- `transition.summary` 过渡废话：综上所述、由此可见、换句话说、本质上、核心在于。
- `attribution.vague` 无源引用：研究表明、数据显示、有专家指出、据报道。
- `cliche.uplift` 正能量收尾：让我们拭目以待、未来可期、与其…不如积极拥抱。
- `sycophantic` 谄媚 / 助手腔：好问题、你说得很对、希望这对你有帮助、要不要我、稳稳接住你。

新增命名空间（human 桶，默认不刷 agent 视图，用 `--review human` 查看）：
- `jargon.engineer` 工程师 / 调试腔：稳稳兜住、收口、根因、落盘、拍板、揪出来（技术语境合理，误杀率高）。
- `jargon.social` 自媒体腔：保姆级、干货、一文读懂、建议收藏、绝绝子（网络用语，真人也用）。
- `translationese` 翻译腔：对于…而言、在…方面、从…的角度来看、通过…来…、基于…。
- `structure.fragment` 戏剧化碎句：连续极短句制造假力量感（对话/刻意节奏应保留）。

机械痕迹（移植自 avoid-ai-writing 的语言无关检测器，高精度、低误杀）：
- `mechanical.zero-width`（`review: none` / 自动修复）：零宽空格 / 连接符 / BOM 等不可见字符，humanizer 工具常插入来绕过检测；直接删除。
- `mechanical.homoglyph`（`review: human`）：西里尔 / 希腊字母里冒充拉丁字母的同形字。
- `mechanical.placeholder`（`review: agent`）：成稿里残留的模板占位符 `{{...}}` / `[姓名]` / `（此处…）`。
- `mechanical.chatbot-artifact`（`review: agent`）：复制 AI 输出带进来的内部标记，如 `:contentReference`、`oaicite`、Bing 角标、chatgpt utm。

误杀防护（取自 shuorenhua `severity.md` 11 条）写进各规则 `note`：引用原文、术语定义、代码/配置、
特定行业语境、技术报告中的工程术语、系统主语行为等场景即使命中也应保留。v1 容忍误差，不为防护牺牲覆盖。

跳过项：shuorenhua Tier 3 单字常用词（重要/关键/核心/优化/提升…）是「全文密度高才算」，逐次 regex 会大量误杀，
本版不收，留给 Agent 的密度判断或后续 density 引擎。

扫描层面：对 Markdown 文件，`check` / `fix` 默认跳过代码块 / frontmatter / 行内代码 / 链接等结构区域（用 `--scan-all` 关闭），
所以代码、链接里的内容不会被规则误杀；`fix` 也不会改动代码块 / frontmatter 内的内容。

## LLM Rule 语义模式

这些问题依赖全文和语境判断，不能只靠关键词决定。

### 1. 空泛总结段

**特征**：段落结尾突然升高到抽象评价，却没有提供新的具体信息。

- 坏例：这一切都具有重要意义，也让未来充满了更多可能。
- 好例：这次失败提醒团队，预算表上的余量并不能替代真实的交付节奏。

### 2. 语体错位

**特征**：文本突然切换成不符合体裁、叙述视角或人物声音的说明书、公文、营销文、课堂总结语气。

- 坏例：她攥紧门把手。接下来，我们将从三个方面分析她的心理变化。
- 好例：本节将从三个方面说明缓存失效的原因。（技术文档中合理）

### 3. 节奏单调

**特征**：连续句子长度接近、结构相同，段落结尾总是短促拔高或抽象收束。

**判断**：
- 如果节奏重复让文本像模板，应重写。
- 如果重复是诗性、角色口吻或修辞设计，可以保留。

### 4. 过度解释

**特征**：文本反复说明显而易见的因果或情绪，替读者下结论。

- 坏例：他离开了房间。这说明他已经不想继续谈下去了，也意味着这段关系出现了裂痕。
- 好例：他离开房间，门没有关。

### 5. 缺少具体信息

**特征**：句子有判断，但没有事实、行动、对象、时间、结果。

- 坏例：这个方案具有很强的可执行性。
- 好例：这个方案只需要改两张表，不碰支付链路。

### 6. 隐藏行动者

**特征**：抽象名词或事物承担人类动作，隐藏了真正做决定的人。

- 坏例：决策逐渐浮现，团队文化发生转变。
- 好例：产品负责人拍板砍掉两个需求，团队开始每周复盘延期原因。

### 7. 金句感

**特征**：句子像社交媒体摘录、标语或结尾 punchline，但没有推动信息。

- 坏例：真正的成长，从来不是抵达，而是看见。
- 好例：她第二天又来了，这次带着账本。

### 8. 段尾机械升华

**特征**：每段结尾都强行拔高主题、总结意义、抛出愿景。

**判断**：
- 如果升华替代了场景、论证或事实，应修复。
- 如果结尾确实压缩观点或改变情绪方向，可以保留。

## 快速审查评分

每次润色都按 5 个维度评分，每项 1-10 分，总分 50。

| 维度 | 审查问题 |
|------|----------|
| Directness | 是直接陈述，还是先公告、铺垫、绕弯？ |
| Rhythm | 句长、段落、收束方式是否有变化？ |
| Trust | 是否信任读者，避免说教、反复解释和情绪操控？ |
| Authenticity | 是否像具体作者、角色、场景发出的声音？ |
| Density | 是否存在可删掉而信息不变的词句？ |

评分处理：
- **低于 35/50**：重点重写结构和具体性。
- **35-42/50**：局部修订。
- **43/50 以上**：轻微润色，保护作者风格。

## 合理保留的情况

并非所有候选都应该删除：

- **对话口癖**：角色自然会说“其实”“说真的”。
- **技术步骤**：文档中“首先、其次、最后”可以提高可读性。
- **正式报告**：结构提示和章节公告可能必要。
- **讽刺与引用**：文本可能故意模仿模板腔。
- **领域术语**：真实业务语境中，某些黑话可能是准确术语。
- **修辞重复**：节奏重复可能是刻意的文学设计。

## 总结

文本润色的核心问题是：**套路化、缺乏具体信息、节奏单调、过度解释**。

修复原则：
1. **删除不必要的填充**。
2. **直接陈述核心信息**。
3. **用具体事实替代抽象评价**。
4. **写出行动者和动作**。
5. **调整节奏，但保留作者声音**。
